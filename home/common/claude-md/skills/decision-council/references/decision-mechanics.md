# Decision Mechanics

`decision-council` skill が以下を判断するためのルール集:

1. **重みの判定** (3 役 / 5 役 / 7 役 のどれにするか)
2. **アドホック役の追加** (ハイブリッド方式: テーブル → LLM フォールバック)
3. **出力フォーマット** (プロジェクト規約検出のロジック)

---

## 1. 重み判定 — 3 / 5 / 7 役

判定軸は **逆戻り可能性 (reversibility)** の単一軸。Bezos の "two-way doors / one-way doors" の考え方に揃える。

### 判定フロー

```
Q1. この決定を「やってみてダメなら戻す」と気軽に言えるか?
    ├─ Yes → Q2 へ (軽量側を検討)
    └─ No  → Q3 へ (重量側を検討)

Q2. 取り消しコストは「時間でいうと数日以内」か?
    ├─ Yes → 3 役 (軽量)
    └─ No  → 5 役 (標準)

Q3. 取り消しに「実質取り消し不能」または「年単位」かかるか?
    ├─ Yes → 7 役 (重量)
    └─ No  → 5 役 (標準)
```

### 重み別の典型例

| 重み | 役数 | 典型例 |
|------|------|--------|
| 軽 (3 役) | DA + Future Self + Pragmatist | ライブラリ選定 (差し替え容易) / 内部 API のリファクタ / 一時的な仕組み導入 |
| 標 (5 役) | + End User + Steel-Manner | フレームワーク選定 / モジュール分割方針 / コーディング規約変更 / ツール導入 |
| 重 (7 役) | + Pre-mortem + Strategist | アーキテクチャ大転換 / プロダクト撤退・廃止 / 採用・組織変更 / DB スキーマ大改修 / 公開 API 設計 |

### 迷ったら重い方に倒す

「3 か 5 か」「5 か 7 か」で迷ったら **重い方** を選ぶ。

理由:
- 重い決定を軽い役数で扱うと **致命的な見落とし** が起きる
- 軽い決定を重い役数で扱っても **時間とトークンを消費するだけ** で害はない
- 非対称なリスクなので、保守的に倒すのが合理的

ただしユーザーが「軽くやってほしい」と明示している場合はその指示を優先する。

### 判定をユーザーに開示する

判定結果は必ずユーザーに伝える:

```
この決定は「{重み}」と判断しました ({N} 役起動)。

理由: {判定根拠を 1〜2 文で}

問題なければこのまま進めます。重く / 軽くしたい場合はお知らせください。
```

判定理由を明示することで、ユーザーが調整できるようにする。

## 2. アドホック役のマッピング表 (テーブル方式)

決定のテーマを以下の表と照合し、該当があればアドホック役を追加する。**複数該当する場合は全て追加** (上限 3 個まで)。

### マッピング表

| テーマキーワード | 追加する役 | 理由 |
|------------------|------------|------|
| 技術選定 / フレームワーク選定 / ライブラリ選定 | 🛠 Maintainer | 5 年後の保守者視点が抜けると技術的負債になる |
| アーキテクチャ / モジュール分割 / 設計 | 🛠 Maintainer | 後から構造を変えるのは高コスト |
| セキュリティ / 認証 / 権限 | 🛡 Adversary | 攻撃者視点なしで決めると穴が残る |
| 外部公開 / 公開 API / OSS 化 | 🛡 Adversary + 👥 End User | 公開後は変更コストが急増 |
| プライバシー / 個人情報 / コンプラ | 🛡 Adversary | 規制当局視点が必要 |
| 撤退 / 廃止 / 凍結 / 縮退 | 🪦 Historian + 💰 Steward | 過去事例 + 撤退コスト両面 |
| 採用 / 組織 / チーム編成 / 体制 | 👥 New Member | 入って間もない人の視点が決定者から漏れる |
| 移行 / マイグレーション / リプレイス | 🪦 Historian | 過去の似た移行事例から学ぶ |
| 内製 vs 外注 / 買う vs 作る | 💰 Steward | 機会費用・撤退コストの判断 |
| プロセス変更 / ルール導入 / 規約変更 | 👥 New Member + 👥 End User | ルールが守られるかは新メンバーが分かる |
| パフォーマンス / SLA / 信頼性 | 🛠 Maintainer | 運用負荷との両立 |
| データ設計 / スキーマ変更 | 🛠 Maintainer | 移行コストが累積する |

### 表の使い方

1. ユーザーから決定したい内容を聞いたら、上の表のキーワードと照合
2. **複数該当** → 全て追加 (ただし上限 3 個。優先度は表の上から順)
3. **重複役は 1 つにまとめる** (例: Maintainer が 2 回該当しても 1 人)

### 表に該当しない場合 → LLM フォールバック (Router 方式)

該当キーワードがなければ、以下の問いを Claude 自身に投げてアドホック役を 1〜3 個生成する:

```
このテーマで意思決定する際、見落としやすい観点は何か?

1. 「決定者が見えていない立場」は誰か (= 立場軸の追加候補)
2. 「決定者が想像しにくい未来」は何か (= 時間軸の追加候補)
3. 「決定者が触れたくない論点」は何か (= 論理軸の追加候補)

それぞれに対応する役を 1 個ずつ提案 (合計最大 3 個)。役名は具体的に、責務は 2〜3 行で。
```

生成した役は [role-definitions.md](role-definitions.md) のプロンプト設計原則 (5 項目) に従って整形する。

### アドホック役を追加しない選択

以下の場合はアドホック役なしで進める:

- 表に該当せず、LLM が「特に追加すべき視点なし」と判断したとき
- 重み「軽 (3 役)」のとき (アドホックを足すと軽量の意味が薄れる)
- ユーザーが「常駐役だけで」と明示したとき

## 3. 出力フォーマット — プロジェクト規約検出

skill は出力フォーマットを **固定しない**。プロジェクトの既存規約を検出して合わせる。

### 検出ロジック (優先順)

1. **ADR ディレクトリの存在確認** (以下のいずれか):
   - `docs/decisions/`
   - `docs/adrs/`
   - `decisions/`
   - `adrs/`
   - `architecture/decisions/`

2. **ADR があれば既存テンプレートを読む**:
   - `0000-template.md` または `template.md` があればそれに合わせる
   - なければ最新の ADR を 1 つ読んでフォーマットを学習
   - 連番を確認し、次の番号で `NNNN-{kebab-case-title}.md` を提案

3. **ADR がなければプロジェクトの慣行を探す**:
   - `CLAUDE.md` / `README.md` / `CONTRIBUTING.md` に「決定の記録方法」が書かれているか
   - 書かれていればそれに従う

4. **どれもなければプレーンテキスト出力**:
   - ファイル化せず、本文に整形して提示するだけ
   - ユーザーが「ファイルに残したい」と明示したら、ADR を新規導入するか聞く

### プレーンテキスト出力の構造

ファイル化しない場合の本文構造 (推奨):

```markdown
## 決定: {1 行サマリ}

**選択肢**:
- A: {内容}
- B: {内容}
- C: {内容} (= 推奨)

**各役の意見**:

| 役 | 推奨 | 主要懸念 |
|----|------|----------|
| 🔥 Devil's Advocate | A | {懸念 1 行} |
| 🕰 Future Self | C | {懸念 1 行} |
| 👥 End User | B | {懸念 1 行} |
| 🧠 Steel-Manner | A | {懸念 1 行} |
| ⚖ Pragmatist | C | {懸念 1 行} |
| (アドホック) | ... | ... |

**意見が割れている論点**:
- {論点 1: 各役がどう違う見解か}
- {論点 2}

**推奨と根拠**: C を推奨。理由は {1〜2 文}。

**この決定で死ぬ条件 (撤退条件)**:
- {条件 1}
- {条件 2}
```

### ADR 形式で出すときの注意

既存の ADR テンプレートを尊重するが、以下の追加要素を必ず入れる:

- **Alternatives Considered**: 各役が支持した選択肢を全て列挙 (どの役が支持したか明記)
- **Devil's Advocate からの反論**: 専用セクションを作る (Consequences の Negative とは別)
- **撤退条件**: Consequences の中または独立セクションで明示

これは「複数役で検証した」事実を ADR に残すための拡張。テンプレート自体は変更しない (本文に追記するだけ)。

## 4. 全体チェックリスト (Step 5 統合時に確認)

司会 (Claude) が統合段階で確認する項目:

- [ ] Devil's Advocate が **何かに反対した** (全肯定だったらやり直し)
- [ ] 役の意見を **要約しすぎていない** (各役 1〜2 行は確保)
- [ ] **意見が割れている論点** を 1 つ以上明示している
- [ ] 推奨を 1 つに絞り、**根拠** を述べた
- [ ] **撤退条件** を 1〜2 個提示した
- [ ] アドホック役の意見も統合に含まれている (アドホックを追加した場合)
- [ ] 出力フォーマットがプロジェクト規約に従っている (ADR 有無を確認した)
