# Interview Questions

`repo-kickoff` skill で使うヒアリング質問集。9 視点 × 主要質問の構造。

各質問には:
- **必須度**: 🔴 必須 / 🟡 推奨 / 🟢 任意
- **質問例**: 実際に投げるテンプレート
- **追加で深掘る場合**: 回答が浅いときの follow-up

ユーザーへのヒアリングは **視点ごとに区切って** 進める。一気に全部投げない。

---

## 視点 1: プロダクト・事業 (Why)

このリポジトリが「なぜ存在するか」を言語化する。最も重要な視点。

### 1.1 存在意義 / ミッション 🔴

**質問**: このリポジトリは一言で言うと何のために存在しますか？（10 秒で説明するなら）

**深掘り**:
- もしこのリポジトリがなかったら、誰がどう困りますか？
- 似た目的の既存ツール・リポジトリと比べて、何が違いますか？

### 1.2 解決する課題 (Problem Statement) 🔴

**質問**: このリポジトリが解決する具体的な課題は何ですか？

**深掘り**:
- その課題は今どうやって回避・対処されていますか？
- 課題の影響を受けているのは誰で、頻度はどれくらいですか？

### 1.3 想定ユーザー / ステークホルダー 🟡

**質問**: 主なユーザーは誰ですか？（自分だけ / チーム / 社内 / OSS など）

**深掘り**:
- ユーザーの技術レベルはどれくらいを想定していますか？
- ユーザー以外で、このリポジトリの存在に影響を受ける人はいますか？

### 1.4 既存の代替手段 🟢

**質問**: 既存の代替手段（OSS、SaaS、自作）はありますか？それらの何が不足していますか？

### 1.5 価値仮説 🟢

**質問**: ユーザーがこれを使うことで、どれくらい時間・労力・コストが節約されると見込みますか？

## 視点 2: スコープ (What) — 最重要

「やること」より「**やらないこと**」を明示することが、長期的な健全性を保つ。

### 2.1 やること (In Scope) 🔴

**質問**: このリポジトリで「やること」を箇条書きで挙げてください（最大 5 個）

### 2.2 やらないこと (Out of Scope) 🔴

**質問**: 逆に、絶対に「やらないこと」「対象外」とすることは何ですか？

**深掘り（重要）**:
- 似ているけど別リポジトリ / 別ツールに任せる領域はありますか？
- 「将来やるかも」ではなく、「将来も絶対やらない」と言い切れることはありますか？
- ユーザーから要望が来そうだけど断ろうと思っている機能はありますか？

### 2.3 隣接プロジェクトとの境界 🟡

**質問**: 関連する他のリポジトリ・サービスはありますか？それらとの境界はどう引きますか？

### 2.4 将来のスコープ拡張余地 🟢

**質問**: 今は対象外だが、将来的に検討する可能性のある領域はありますか？

---

## 視点 3: エンジニアリング (How)

### 3.1 技術スタック 🟡

**質問**: 主要な言語・フレームワーク・ランタイムは？決定済み / 未定どちらでも OK

### 3.2 アーキテクチャ概要 🟡

**質問**: 全体構成を一行で表現するなら？（例: 「Next.js + Prisma の単一サービス」「CLI ツール、依存は Node のみ」）

**深掘り**:
- 主要な構成要素（モジュール / サービス / 層）は？
- データの流れは？

### 3.3 主要な設計判断 🟢

**質問**: 「これは敢えてこう選んだ」という設計判断はありますか？（後で ADR にする種）

**例**:
- 「DB は Postgres ではなく SQLite。理由: ローカル完結を優先」
- 「フレームワークは使わず素の Node。理由: 依存を最小化」

### 3.4 外部依存 🟢

**質問**: 外部サービス・API・有料 SaaS への依存はありますか？

## 視点 4: 品質・運用 (Quality)

### 4.1 非機能要件 🟡

**質問**: 性能・可用性・セキュリティで「ここは外せない」基準はありますか？

**例**:
- 「レスポンス 200ms 以内」「99.9% 稼働率」「個人情報を扱うので暗号化必須」

### 4.2 テスト戦略 🟢

**質問**: テストはどこまでやる予定ですか？（単体 / 結合 / E2E、カバレッジ目標など）

### 4.3 監視・観測 🟢

**質問**: 本番運用する場合、何をどう監視しますか？（ログ・メトリクス・アラート）

### 4.4 インフラ・デプロイ 🟢

**質問**: どこにデプロイしますか？（自分の PC / VPS / Vercel / k8s など）

---

## 視点 5: ロードマップ (When)

### 5.1 MVP の定義 🔴

**質問**: 「最小限これができれば MVP として価値がある」という機能セットは？

**深掘り**:
- MVP に **入れない** 機能は？（視点 2 と関連）
- MVP の完成目標はいつ頃ですか？

### 5.2 フェーズ分け 🟡

**質問**: MVP の後、Phase 2, 3 として想定している段階はありますか？

**フォーマット例**:
```
Phase 1 (MVP): 〜まで動く
Phase 2: 〜を追加
Phase 3: 〜まで拡張
```

### 5.3 各フェーズの完了条件 (Definition of Done) 🟡

**質問**: 各フェーズが「完了した」と判断する基準は何ですか？

### 5.4 想定タイムライン 🟢

**質問**: 大まかなタイムラインはありますか？（厳密でなく OK）

## 視点 6: 成功・撤退 (Success / Sunset) — 見落としがち

成功と「終わり」を最初に言語化する。これがないと永遠に続いて負債化する。

### 6.1 成功指標 (KPI / Success Metrics) 🟡

**質問**: このリポジトリが「成功した」と判断する指標は？（数値 / 定性的どちらでも）

**例**:
- 「自分が毎日使っている」「チームの 80% が使っている」「GitHub Star 100」「年間 10 時間節約」

### 6.2 完了条件 🟡

**質問**: このリポジトリが「役目を終えた」と判断する条件はありますか？

### 6.3 撤退条件 / 凍結条件 🔴

**質問**: 逆に、「これが起きたら撤退・凍結する」という条件は？

**深掘り**:
- 6 か月使われなかったら？
- メンテコストが価値を上回ったら？
- 上位互換のツールが登場したら？

### 6.4 後継への移行パス 🟢

**質問**: 撤退時、ユーザーやデータはどこに移行しますか？

## 視点 7: コラボレーション (Who)

### 7.1 メンテナ / オーナー 🟡

**質問**: 誰がメンテナンスしますか？（自分一人 / チーム / 部署）

### 7.2 貢献ポリシー 🟢

**質問**: OSS の場合、外部からの貢献を受け入れますか？どんなポリシーで？

### 7.3 コミュニケーションチャネル 🟢

**質問**: 議論・質問はどこでやりますか？（Issue / Slack / Discord / なし）

### 7.4 レビュー体制 🟢

**質問**: コードレビューはどう運用しますか？（PR 必須 / 直接 push / セルフレビュー）

---

## 視点 8: リスク・制約 (Risks)

### 8.1 技術的リスク 🟢

**質問**: 「これがうまくいかないと困る」技術的な不確実性はありますか？

### 8.2 法的・コンプライアンス制約 🟢

**質問**: ライセンス・個人情報・社内規定など、守るべき制約はありますか？

### 8.3 予算・リソース制約 🟢

**質問**: 月額コスト上限・使える時間など、リソース面の制約はありますか？

### 8.4 前提と不確実性 🟢

**質問**: 「これが前提」「これは未確定」という事項はありますか？

## 視点 9: AI 協働 (CLAUDE.md 特有)

このリポジトリで AI（Claude Code）に作業させる際の文脈情報。CLAUDE.md にのみ書く。

### 9.1 AI に守ってほしい暗黙ルール 🟡

**質問**: AI が初見でも従ってほしいルールはありますか？

**例**:
- 「コミットメッセージは日本語」「PR は作らず main に直 push」「テストは必ず書く」

### 9.2 ドメイン用語 / 略語 🟡

**質問**: このプロジェクト固有の用語・略語はありますか？

**例**:
- 「`profile` = nix の構成プロファイル」「`kickoff` = 新規リポジトリ立ち上げ作業」

### 9.3 触れてほしくない領域 🟡

**質問**: AI に勝手に触ってほしくないファイル・ディレクトリはありますか？

**例**:
- 「`migrations/` 配下は手動で触る」「`secrets/` は読まない」

### 9.4 プロジェクト特有のワークフロー 🟢

**質問**: 一般的でない作業手順はありますか？（特殊なビルド・デプロイ・テスト方法など）

## ヒアリング進行のコツ

1. **必須項目（🔴）から聞く** — 視点 1, 2, 5, 6 の 🔴 を先に
2. **回答を要約して確認** — 「つまり〜という理解で合ってますか？」
3. **「未定」を恐れない** — 必須項目以外は `TBD` と書いて先に進む
4. **答えにくい質問は具体例を示す** — 質問だけ投げず「例: 〜のような」と添える
5. **既存情報があれば再質問しない** — README.md などから読み取れる項目はスキップ

---

## 必須項目の充足判定（生成前チェック）

ヒアリング後、ファイル生成に進む前に **必須項目（🔴）の充足率** を確認する。

### 必須項目一覧

| # | 視点 | 質問 |
|---|------|------|
| 1.1 | プロダクト | 存在意義 / ミッション |
| 1.2 | プロダクト | 解決する課題 |
| 2.1 | スコープ | やること |
| 2.2 | スコープ | **やらないこと** |
| 5.1 | ロードマップ | MVP の定義 |
| 6.3 | 成功・撤退 | **撤退条件 / 凍結条件** |

合計 **6 項目**。

### 判定ルール

```
充足率 = 内容のある回答が得られた必須項目数 / 6
```

- **充足率 < 50%（3 項目未満）**: 生成を保留する
  - ユーザーに「このまま生成すると `TBD` だらけになり、後で誰も読まない置物になります」と伝える
  - 不足している項目を提示し、追加で考えてみるよう促す
  - 「どうしても今は答えられない」とユーザーが言えば、その理由をメモして生成に進む（ユーザーの判断を尊重）

- **充足率 >= 50%（3 項目以上）**: 生成に進む
  - 未充足項目は `TBD（次回更新時に決定）` と明示してファイルに含める

### 「内容のある回答」の判定

以下は **充足とみなさない**:
- 一行で内容のない返答（例: 「特にないです」「考えてません」）
- 質問の言い換えに過ぎない返答（例: 「課題は何？」→「色々あります」）
- 「適当に書いておいて」と AI に丸投げ

以下は **充足とみなす**:
- 短くても具体的（例: 「個人ブログ用の静的サイト生成器」）
- 「現時点では未定だが、〜の方向で考えている」のような暫定方針
